01 Process-to-Process Delivery & Client-Server Paradigm ▾

Delivery Types in TCP/IP

Node-to-Node (Layer 2)

Data Link Layer. Between two directly connected devices. Uses MAC address.

Host-to-Host (Layer 3)

Process-to-Process (Layer 4)

Transport Layer. Between specific running apps on two hosts. Uses port numbers.

Client-Server Paradigm

🖥️ Server
  • Always-on, fixed IP address
  • Listens on a well-known port
  • Serves multiple clients simultaneously
  • Passive — waits for requests
  • Example: Web server (port 80)
💻 Client
  • Communicates with server when needed
  • May have dynamic IP address
  • Uses ephemeral port (assigned by OS)
  • Active — initiates connection
  • Example: Browser requesting a page

02 UDP — User Datagram Protocol ▾

UDP is a connectionless, unreliable transport protocol providing process-to-process communication with minimum overhead. Defined in RFC 768.

🧠 UDP = "Uncle Doesn't Promise" — no delivery guarantee, no ordering, no error recovery

UDP Header (8 bytes — Fixed)

Source Port
16 bits
Dest Port
16 bits
Length
16 bits
Checksum
16 bits
UDP Header = 8 bytes (fixed) · Length = Header + Data (minimum value = 8)

Features:

  • No connection setup / teardown
  • No ACK — fire and forget
  • No flow or congestion control
  • Supports broadcast and multicast
  • Very low latency
  • Optional checksum (detection only)

When to use UDP:

  • Real-time apps (VoIP, video calls)
  • Live streaming
  • DNS queries (small and fast)
  • DHCP, SNMP, TFTP, RIP
  • Online gaming
  • App handles its own reliability
⚠️ UDP header is only 8 bytes. TCP header is minimum 20 bytes. Very common exam question.
03 TCP — Transmission Control Protocol & Flow Control ▾

TCP is a connection-oriented, reliable, full-duplex protocol. Defined in RFC 793. Provides byte-stream service.

🧠 TCP = "Totally Controlled Pipeline" — reliable, ordered, error-checked with flow & congestion control

TCP Header (Minimum 20 bytes)

Source Port (16 bits)
Destination Port (16 bits)
Sequence Number (32 bits)
Acknowledgement Number (32 bits)
HLEN(4) · Reserved(6) · Flags(6)
Window Size (16 bits)
Checksum (16 bits)
Urgent Pointer (16 bits)
Options (variable) + Padding
TCP Flags (6 bits): URG · ACK · PSH · RST · SYN · FIN
Memory trick: "Until Amazing People Remember Sending Forms"

TCP 3-Way Handshake (Connection Setup)

Client
──── SYN (seq=x) ────→
Server
Client
←── SYN+ACK (seq=y, ack=x+1) ──
Server
Client
──── ACK (ack=y+1) ────→
Server
🧠 3-way handshake: SYN → SYN-ACK → ACK
Like a phone call: "Hello?" → "Hello, who's there?" → "It's me, let's talk!"

TCP 4-Way Termination (Connection Teardown)

Client → FIN → Server  |  Server → ACK → Client  |  Server → FIN → Client  |  Client → ACK → Server

4 steps because each direction closes independently. Client enters TIME_WAIT state (2×MSL) after final ACK.

TCP Flow Control — Sliding Window

Sender can send: min(cwnd, rwnd) bytes without waiting for ACK
rwnd (Receive Window)

Advertised by receiver in TCP header. Based on available buffer space. Prevents receiver overflow.

cwnd (Congestion Window)

Maintained by sender. Based on network congestion. Controlled by slow start & congestion avoidance.

🧠 Flow Control vs Congestion Control:
Flow = Receiver tells sender to slow down (buffer problem)
Congestion = Network tells sender to slow down (router queue problem)

TCP Congestion Control Phases

1. Slow Start
  • cwnd starts at 1 MSS
  • Doubles every RTT (exponential)
  • Until cwnd ≥ ssthresh
2. Congestion Avoidance
  • cwnd increases +1 per RTT (linear)
  • After cwnd hits ssthresh
  • Until timeout or 3 dup ACKs
3. Fast Retransmit
  • 3 duplicate ACKs → loss detected
  • ssthresh = cwnd/2
  • cwnd = ssthresh (TCP Reno)

04 FTP — File Transfer Protocol ▾
FTP transfers files between client and server. Uses two separate TCP connections. Defined in RFC 959. Works at Application Layer.
🧠 "21 Controls, 20 Delivers" → Port 21 = Control (commands) · Port 20 = Data (file transfer)
📋 Control Connection — Port 21
  • Always open (persistent throughout session)
  • Sends commands: USER, PASS, LIST, RETR, STOR
  • Uses ASCII text commands
  • Also called "command channel"
📁 Data Connection — Port 20
  • Opens only when transferring files
  • Closes after each file transfer
  • ASCII mode (text) or Binary mode
  • Also called "data channel"

Active vs Passive FTP

FeatureActive ModePassive Mode
Who opens data connection?Server connects to clientClient connects to server
Server data portPort 20Random high port (>1023)
Firewall friendly?No (server initiates inbound)Yes (client initiates both)
Common useLegacy systemsModern internet (preferred)
FTP Variants
  • FTPS: FTP + SSL/TLS (secure)
  • SFTP: SSH File Transfer Protocol
  • TFTP: Trivial FTP — UDP, no auth, simpler
FTP Features
  • Anonymous FTP (no login needed)
  • File types: ASCII, Binary
  • Commands: USER, PASS, RETR, STOR, LIST, QUIT
⚠️ FTP sends credentials in plaintext → NOT secure. SFTP/FTPS used in modern secure environments.
05 SMTP — Simple Mail Transfer Protocol ▾
SMTP is used to send emails between mail servers and from client to server. Port 25 (server-to-server) or 587 (client-to-server). Uses TCP.
🧠 SMTP = "Send Mail To People" — only for SENDING.
Receiving uses POP3 (port 110) or IMAP (port 143).

SMTP Key Characteristics:

  • Push protocol — sender pushes mail to server
  • Uses TCP (reliable delivery)
  • Originally 7-bit ASCII text only
  • MIME adds binary/multimedia support
  • Commands: HELO, MAIL FROM, RCPT TO, DATA, QUIT
  • Codes: 220 Ready · 250 OK · 354 Start · 550 Unknown user

06 Telnet — Remote Login Protocol ▾
Telnet (TELecommunication NETwork) allows a user to log into a remote computer and use it as if sitting locally. Port 23, uses TCP.
🧠 Telnet = "Type Everywhere, Login Network" — remote terminal access over network. The original SSH (but insecure).
⚙️ How Telnet Works
  • Client connects to server on port 23
  • Server presents login prompt
  • User types commands → sent to remote machine
  • Remote machine executes → sends output back
  • Uses NVT (Network Virtual Terminal) concept
  • NVT = standard character representation for transmission
⚠️ Limitations
  • Sends all data as plain text — no encryption
  • Username & password visible on network
  • Vulnerable to eavesdropping/sniffing
  • Replaced by SSH (Secure Shell) on port 22
  • Still used for testing/debugging network devices

Telnet Options Negotiation

Client and server negotiate capabilities using 4 commands:

WILL

Sender wants to enable option

WONT

Sender refuses to enable option

DO

Sender wants receiver to enable

DONT

Sender wants receiver to disable

⚠️ Telnet = insecure (plain text). SSH = secure (encrypted). In exams: Telnet = "insecure remote login".
07 WWW, HTTP & URL ▾

WWW — World Wide Web

A system of interlinked hypertext documents accessed via the Internet. Created by Tim Berners-Lee in 1989 at CERN. NOT the same as the Internet.

HTML

HyperText Markup Language — structure & content of web pages

HTTP

HyperText Transfer Protocol — how browsers & servers communicate

URL

Uniform Resource Locator — address of a resource on the web

🧠 WWW ≠ Internet: Internet = physical infrastructure (cables, routers) · WWW = one service running ON the internet
HTTP is a stateless, application-layer protocol for transferring web pages. Port 80 (HTTP) / Port 443 (HTTPS). Uses TCP.

HTTP Request Methods

GET

Retrieve data. Parameters in URL. Safe & idempotent.

POST

Send data. Body payload. Forms, login. NOT idempotent.

PUT

Update resource entirely. Idempotent.

DELETE

Remove a resource. Idempotent.

HEAD

Like GET but headers only, no body.

OPTIONS

Ask server which methods are supported.

08 DNS — Domain Name System ▾
DNS translates domain names (www.google.com) → IP addresses (142.250.190.4). Port 53. Uses UDP for queries and TCP for zone transfers.
🧠 DNS = "Domain Name to IP System" — the phonebook of the Internet.
You know the name, DNS finds the number (IP address).

DNS Hierarchy (Inverted Tree)

. (Root) ┌──────┼──────┐ .com .org .in ← Top Level Domain (TLD) ┌────┘ google.com ← Second Level Domain ┌────┘ www.google.com ← Subdomain / FQDN

Types of DNS Servers:

  • Root Name Server: Top of hierarchy. 13 sets (A–M). Knows all TLD servers.
  • TLD Name Server: Handles .com, .org, .in etc. Knows authoritative servers.
  • Authoritative Name Server: Has actual DNS records. Final answer.
  • Recursive Resolver: ISP's DNS. Does all the work on your behalf.
  • Local DNS Cache: Browser and OS cache DNS for speed.

DNS Record Types:

RecordPurpose
ADomain → IPv4 address
AAAADomain → IPv6 address
MXMail Exchange server
CNAMEAlias → canonical name
NSName Server record
PTRIP → domain (reverse lookup)
TXTText info (SPF, DKIM)

DNS Resolution Process

1. Browser checks local cache → found? Done!
2. Query sent to Local DNS Resolver (ISP / recursive)
3. Resolver checks its cache → found? Done!
4. Resolver queries Root Name Server → returns TLD server address
5. Resolver queries TLD Name Server (e.g. .com) → returns authoritative server
6. Resolver queries Authoritative Name Server → returns final IP
7. Resolver caches result → returns IP to browser
8. Browser connects to IP → website loads
Recursive Query

Resolver does ALL work. Client gets final IP. Most common for client → resolver.

Iterative Query

Server returns best referral. Client queries each server itself. Used between servers.

🧠 DNS resolution order — "Brave Rascals Run To Answer"
Browser cache · Resolver · Root · TLD · Authoritative
⚠️ DNS uses UDP port 53 for queries · TCP port 53 for zone transfers. Very common exam question.
09 DHCP & BOOTP ▾
DHCP automatically assigns IP addresses and network config to devices. Server port 67, Client port 68. Uses UDP.
🧠 DHCP = "Dynamic Host Configures Please"
Automatically gives IP, subnet mask, gateway, DNS to any device joining the network.

DHCP 4-Step Process — DORA

D — Discover

Client broadcasts to find DHCP server
(src=0.0.0.0, dst=255.255.255.255)

→
O — Offer

Server offers an IP address with lease time to client

→
R — Request

Client broadcasts acceptance of offered IP address

→
A — Acknowledge

Server confirms; client may now use the IP address

🧠 "Devices Often Request Addresses" → DORA
Discover · Offer · Request · Acknowledge
What DHCP provides
  • IP address (dynamic or reserved)
  • Subnet mask
  • Default gateway
  • DNS server address
  • Lease duration
DHCP Lease
  • IP given for limited time (lease)
  • Client renews at T/2 of lease time
  • Fails → retry at T×7/8
  • Expired lease → redo full DORA
  • Reserved IP = MAC bound to IP

BOOTP — Bootstrap Protocol

🥾 BOOTP (Bootstrap Protocol) — Predecessor to DHCP

Used to boot diskless workstations over a network. Also uses UDP ports 67/68.

FeatureBOOTPDHCP

🧠 BOOTP = "Boot Only One Time Parameters" (static, old)
DHCP = Dynamic, modern, evolved from BOOTP. Backward compatible.
⚠️ Both BOOTP and DHCP use UDP ports 67 (server) and 68 (client). DHCP evolved FROM BOOTP with dynamic allocation added.

Top 8 Exam Traps

1. UDP header = 8 bytes fixed · TCP header = minimum 20 bytes (up to 60 bytes with options)
2. DNS uses UDP for queries (port 53) but TCP for zone transfers. Not purely UDP!
3. SMTP is for sending only. POP3/IMAP are for receiving/accessing. Very commonly confused.
4. FTP uses two TCP connections — port 21 (control) and port 20 (data). HTTP uses only one.
5. HTTP is stateless. Cookies/sessions are used to add state on top of HTTP.
6. WWW ≠ Internet. Internet = network infrastructure. WWW = one service running on it.
7. Telnet = insecure plain text (port 23). SSH = secure encrypted (port 22). Always distinguish.
8. DHCP evolved from BOOTP. Both use ports 67/68 UDP. DHCP added dynamic IPs + lease concept.
